## 它的作用

`triage` 将项目工单跟踪器中的问题通过一个小的**状态机**进行分类处理——对其进行分类、验证诉求、必要时盘问以完善问题、然后留下一个可供代理处理的简要说明。

它从不盲目贴标签。每个经过分类的项目都带有恰好一个**类别**标签（`bug` / `enhancement`）和恰好一个**状态**标签（`needs-triage`、`needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix`），并且它会*推荐并等待*——它会告诉你其分类和状态判断及理由，然后只在你的指示下行动。在任何内容被提升为 `ready-for-agent` 之前，它首先验证诉求：缺陷会被复现，PR 会被拉取并运行。

## 何时使用

你通过键入 `/triage` 并用自然语言描述你想要什么来调用它——代理不会主动使用它。"显示任何需要我关注的内容"、"我们来看看 #42"、"将 #42 移至 ready-for-agent"。

当你的工单跟踪器中有原始、未经评估的报告，并且你希望它们被分类、验证并转化为代理或人类可以接手的工作时使用。要将一个已确定的对话转化为一份新的规范，使用 [to-prd](https://aihero.dev/skills-to-prd)；要将现有的 PRD 拆分为工单，使用 [to-issues](https://aihero.dev/skills-to-issues)。`triage` 是反向的方向——它处理*已经*存在于跟踪器中的内容。

## 前置条件

`triage` 读取并写入你的工单跟踪器，因此 [setup-matt-pocock-skills](https://aihero.dev/skills-setup-matt-pocock-skills) 必须首先配置好跟踪器和标签映射。上述标签名称是**规范化的**——你跟踪器中的实际标签字符串可能有所不同，而映射正是由设置提供的。配置还决定了外部 PR 是否算作请求面，以及谁算作外部人员。

## PR 是附带了代码的问题单

如果跟踪器将外部拉取请求视为请求面，`triage` 会通过*相同的*状态机来处理它们：相同的类别标签、相同的状态、相同的转换——只是状态是根据差异（diff）而不是报告来判定的。`ready-for-agent` 意味着附带了简要说明，并且代理应对代码采取下一步行动；`ready-for-human` 意味着可以合并了。发现表面仅显示外部 PR，但一个明确指名的 PR 无论作者是谁都会被分类处理。

## 在给出简要说明前先验证

使 `triage` 区别于随意贴标签的步骤是验证。它根据报告人的步骤复现缺陷，或者拉取 PR 并运行测试，然后报告：已确认（附代码路径）、已失败、或细节不足（这本身就是一个强烈的 `needs-info` 信号）。它还进行两项代码库检查——**冗余性**（这个功能是否已经实现了？如果是，则为 `wontfix`）和**先前拒绝**（`.out-of-scope/` 是否已经否定了它？）。经过验证的确认会产生一个强得多的代理简要说明；猜测则不会。

## 它发挥作用的表现

- 它接触的每个项目最终都恰好带有一个类别标签和一个状态标签——绝不是零个，也绝不是两个冲突的状态。
- 它会向你提供带有理由的推荐并等待，而不是自行重新贴标签。
- 缺陷在被标记为 `ready-for-agent` 之前会被复现，PR 会被运行。
- 它发布到跟踪器的每条评论都以 `> *This was generated by AI during triage.*` 免责声明开头。

## 它的定位

`triage` 是对你工单跟踪器的**定期维护**——在报告堆积时运行它，以保持队列有序和 `ready-for-agent` 列的可信度。它位于跟踪器的最前端，处于构建链的上游：它编写的简要说明正是 [tdd](https://aihero.dev/skills-tdd) 后续进行实现时需要读取的内容。当请求需要打磨时，它依赖 [grilling](https://aihero.dev/skills-grilling) 和 [domain-modeling](https://aihero.dev/skills-domain-modeling) 一次一个问题地将其盘问成形。它的近邻是 [to-prd](https://aihero.dev/skills-to-prd)，后者从新的对话中填充跟踪器，而 `triage` 处理已经存在的内容。当你不确定适合哪种技能或流程时，[ask-matt](https://aihero.dev/skills-ask-matt) 会为你指引方向。
