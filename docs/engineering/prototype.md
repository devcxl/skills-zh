## 功能

`prototype` 构建一个小型、可丢弃的程序，其唯一职责就是回答一个设计问题——这个状态模型感觉对吗，或这个 UI 应该长什么样。

代码**从第一天起就是可丢弃的**，并被标记为如此。它不带任何测试，不带任何超出运行所需之外的错误处理，没有抽象，没有持久化。关键是快速学到东西然后删除它——因此当你开始加固它时，你已经停止原型设计了。

## 何时使用

输入 `/prototype`，或者当任务合适时 Agent 会自动调用。

当你有一个在纸上难以确定的设计问题时使用——一个你无法在头脑中装下的状态机，或一个你不到看到几个版本并排就想象不出的界面。如果相反，已经构建好的东西行为不正确需要找出原因，使用 [diagnosing-bugs](https://aihero.dev/skills-diagnosing-bugs)；原型设计探索*要构建什么*，而不是*已构建的东西哪里坏了*。

## 两个分支

问题决定形状，有两种形状：

- **"这个逻辑/状态模型感觉对吗？"**——一个小的交互式终端应用，将状态机推过尴尬的情况，在每个动作后打印完整状态，以便你可以观察什么发生了变化。
- **"这应该长什么样？"**——同一条路由上几种截然不同的 UI 变体，可以从浮动栏切换，以便你比较真实的渲染效果而非凭空想象。

选错分支会浪费整个原型，因此先有问题是关键。两个分支都将状态保存在内存中，从一个命令运行，并在每一步显示完整状态。

## 答案才是制成品

代码是可丢弃的；**答案**是唯一值得保留的东西。当原型已经解决了问题，将结论捕获到某个持久的地方——一条 commit 消息、一个 ADR、一个 issue，或旁边的 `NOTES.md`——连同它回答的问题一起，然后删除或吸收代码。一个在仓库中腐烂的原型已经超越了其存在的目的。

## 在生态中的位置

`prototype` 是一个随时可用的独立技能：你切入它来解析一个设计问题，然后退出。它的答案通常输入下一步——一个已验证的状态模型或 UI 方向成为 [to-prd](https://aihero.dev/skills-to-prd) 的确定输入以供编写，或一个值得通过 [domain-modeling](https://aihero.dev/skills-domain-modeling) 记录的架构决策。当你不确定哪个技能或流程适合时，[ask-matt](https://aihero.dev/skills-ask-matt) 会引导你。
