# 编写技能

## 流程

1. **收集需求** — 询问用户：
   - 该技能覆盖什么任务/领域？
   - 应该处理哪些具体用例？
   - 是否需要可执行脚本，还是只需要指令？
   - 需要包含哪些参考资料？

2. **起草技能** — 创建：
   - 包含简明指令的 SKILL.md
   - 如果内容超过 500 行，添加额外的参考文件
   - 如果需要确定性操作，添加工具脚本

3. **与用户审查** — 展示草稿并询问：
   - 这覆盖了你的用例吗？
   - 有什么遗漏或不清晰的吗？
   - 哪个部分需要更多/更少的细节？

## 技能结构

```
skill-name/
├── SKILL.md           # 主指令（必需）
├── REFERENCE.md       # 详细文档（按需）
├── EXAMPLES.md        # 使用示例（按需）
└── scripts/           # 工具脚本（按需）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力简述。在 [特定触发条件] 时使用。
---

# 技能名称

## 快速开始

[最小可行示例]

## 工作流

[包含检查清单的分步流程，用于复杂任务]

## 高级功能

[链接到单独文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**你的智能体在选择加载哪个技能时看到的唯一内容**。它会和其他已安装的技能一起出现在系统提示中。你的智能体读取这些描述，根据用户的请求选择相关技能。

**目标**：给智能体刚好足够的信息来知道：

1. 这个技能提供什么能力
2. 何时/为何触发它（特定的关键词、上下文、文件类型）

**格式**：

- 最多 1024 个字符
- 用第三人称写
- 第一句：它做什么
- 第二句："在 [特定触发条件] 时使用"

**好的示例**：

```
从 PDF 文件中提取文本和表格、填写表单、合并文档。在处理 PDF 文件或用户提到 PDF、表单或文档提取时使用。
```

**坏的示例**：

```
帮助处理文档。
```

坏的示例让智能体无法将其与其他文档类技能区分开来。

## 何时添加脚本

在以下情况添加工具脚本：

- 操作是确定性的（验证、格式化）
- 同样的代码会被反复生成
- 错误处理需要显式定义

脚本能节省 token 并提高相对于生成代码的可靠性。

## 何时拆分文件

在以下情况将内容拆分为单独文件：

- SKILL.md 超过 100 行
- 内容有明显的领域划分（如金融 vs 销售模式）
- 高级功能很少用到

## 审查清单

起草完成后，验证：

- [ ] 描述包含触发条件（"在……时使用"）
- [ ] SKILL.md 在 100 行以内
- [ ] 没有对时间敏感的信息
- [ ] 术语一致
- [ ] 包含具体示例
- [ ] 引用深度不超过一级
