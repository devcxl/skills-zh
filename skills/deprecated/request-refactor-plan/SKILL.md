---

此技能将在用户想要创建重构请求时被调用。你应该执行以下步骤。如果认为没有必要，可以跳过某些步骤。

1. 向用户询问他们想要解决的问题的详细描述，以及任何潜在的解决方案构思。

2. 探索仓库以验证他们的断言并理解代码库的当前状态。

3. 询问他们是否考虑过其他选项，并向他们展示其他选项。

4. 就实现方案采访用户。要极其详细和彻底。

5. 敲定实现的精确范围。确定你计划改变什么，以及不改变什么。

6. 在代码库中检查此区域的测试覆盖情况。如果测试覆盖不足，询问用户的测试计划是什么。

7. 将实现拆解为由微小 commit 组成的计划。记住 Martin Fowler 的建议："让每个重构步骤尽可能小，这样你始终能看到程序在正常工作。"

8. 创建一个包含重构计划的 GitHub issue。使用以下模板作为 issue 描述：

<refactor-plan-template>

## 问题陈述

开发者面临的问题，从开发者的视角描述。

## 解决方案

问题的解决方案，从开发者的视角描述。

## Commits

一份长而详细的实现计划。用通俗的英文撰写计划，将实现拆解为尽可能最小的 commit。每个 commit 都应让代码库处于可工作的状态。

## 决策文档

一份已做出的实现决策列表。可以包括：

- 将要构建/修改的模块
- 这些模块中将被修改的接口
- 来自开发者的技术澄清
- 架构决策
- Schema 变更
- API 契约
- 具体交互

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

一份已做出的测试决策列表。包括：

- 什么是好的测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的参考先例（即代码库中类似类型的测试）

## 范围之外

对本次重构不涉及的内容的描述。

## 补充说明（可选）

关于重构的任何进一步说明。

</refactor-plan-template>
