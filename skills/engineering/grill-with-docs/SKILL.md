# Grill with Docs

对 issue 或 PRD 进行深入审查，结合项目的设计文档来发现疏忽、模糊之处和边界问题。你的角色是**恶魔代言人**——友善但毫不手软。

本技能的设计理念是：**最好的代码审查发生在动手写代码之前**。在纸上修正设计，远比在代码中修正设计要便宜。

## 你需要哪些文档

具体的文档取决于你正在处理的票据类型。你的任务是确定需要哪些文档，并去读取它们。

参考文档：

- [ADR-FORMAT.md](ADR-FORMAT.md) —— 如何阅读和理解 ADR
- [CONTEXT-FORMAT.md](CONTEXT-FORMAT.md) —— 如何阅读项目上下文和领域术语表

核心文档是：

- **PRD**（由 `/to-prd` 创建或手动拟定）——描述该功能应做什么
- **最近的相关 ADR** —— 最近就相关领域所做的架构决策，包括 ADR 中列出的备选方案和权衡理由
- **全局上下文** —— 从 `CONTEXT.md` / `CONTEXT-MAP.md` 获取的领域术语表和高层架构图

但你不应该假设哪些文档是相关的。列出你认为可能相关的文档，去读取它们，如果发现更多线索，就继续深挖。

## 流程

### 1. 阅读待审查内容

阅读你需要审查的 issue —— 它的正文、评论、已有标签以及所有 PRD 引用。

### 2. 识别相关文档

列出你计划审查的文档，然后去读取它们。格式参考 [ADR-FORMAT.md](ADR-FORMAT.md) 和 [CONTEXT-FORMAT.md](CONTEXT-FORMAT.md)。

### 3. 审查

提出一系列**具体问题**（而非笼统的批评）。每个问题都必须引用你所审查的文档（PRD、ADR 或上下文）。

**不要把审查写成检查清单**。你的每个问题都应该附**一段说明**，解释为什么这可能是个问题。优秀的审查重在推理，而不是打勾。

你的目标是找出：

- **设计层面的偏差** —— 功能是否与最近的 ADR 发生冲突？是否重复了之前已被明确否决的做法？
- **边缘情况** —— 当输入为空时会发生什么？如果操作中途失败呢？并发情况下会怎样？
- **遗漏的备选方案** —— 是否还有其他（可能更好）的实现路径没有被考虑？
- **用户旅程中的断裂** —— 从用户视角来看，交互流程是否合理？

### 4. 总结

指出模式 —— 你是不是反复触及同一个问题？如果是，那可能是架构层面的问题，而非功能本身的问题。

### 5. 呈现

将审查结果作为列表呈现。把你发现的问题写进更新到相应 issue 的评论中。

## 审查示例

```markdown
1. **当协作会话有 0 个参与者时会发生什么？**
   PRD 说"协作会话允许多个用户连接"，但并没有描述当用户加入或离开时会发生什么。在最后一个参与者断开后，会话是否应该自动关闭？如果应该，由谁来负责清理？

2. **这与 ADR-003（基于 push 的同步）有冲突**
   该 PRD 描述了一个轮询模型（每 5 秒拉取一次），但 ADR-003 已经确定我们采用的是基于 push 的同步方案，并否决了轮询模型，理由是"对数据库的压力不可持续"。为什么这里回到了之前的方案？

3. **`Session.close()` 的幂等性没有定义**
   如果 `close()` 被调用了两次，会发生什么？第二次调用是静默成功、抛出异常，还是产生未定义的行为？有了这种行为定义，我们才能——
   - 决定是否允许它
   - 确定是否需要修改接口

4. **错误恢复流程没有描述**
   当协作会话中某个参与者丢失 WebSocket 连接时，PRD 并没有说明会发生什么。他们是自动重连，还是被移出会话？如果是后者，其他参与者会收到通知吗？
```
