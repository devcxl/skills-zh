# ADR 格式

架构决策记录（ADR）存放在 `docs/adr/` 目录中。每个 ADR 都是一个独立的 markdown 文件，记录了一项架构决策及其背景和理由。

> 命名惯例：`NNNN-slug.md`（例如 `0001-use-sqlite.md`）。

## ADR 的内容

ADR 的核心在于说明**为什么选择某个方案**。一个可靠的 ADR 会包含：

- **标题** —— 决策的简要命名
- **状态** —— 已提议 / 已接受 / 已废弃 / 已取代
- **背景** —— 促使做出该决策的问题或约束
- **决策** —— 团队最终选择的方案
- **备选方案** —— 被考虑但未被采纳的方案，以及未采纳的原因
- **后果** —— 该决策带来的影响，包括正面和负面的

## 如何解读一个 ADR

当你阅读一个 ADR 时，问自己：

### 这项决策制定了什么规矩？

ADR 的核心是**决策**本身。注意那些强制性的措辞（如"必须"、"禁止"），以及明确说明的方案选择。

### 为什么这个方案优于其他方案？

阅读**备选方案**部分。团队考虑了哪些其他选择？为什么它们被否决了？这段会告诉你哪些路径是有意不走的——这一点在审查新功能时非常关键。

### 有哪些已知的权衡？

阅读**后果**部分。每一项决策都会带来取舍。了解这些取舍，你就能在审查与该决策相关的新代码时更敏锐地发现潜在的摩擦点。

### 这个 ADR 还是现行有效的吗？

检查**状态**。一个 ADR 可能被更新的 ADR 所取代（状态为 `已取代`，并在顶部标有 superseeding 引用）。如果被取代，你应当查阅取代它的 ADR，以获取最新的决策信息。

## 给子技能的说明

当 `grill-with-docs` 或 `triage` 告诉你"结合 ADR 进行审查"时，即意味着你要：

1. 阅读与该领域相关的近期 ADR
2. 检查提案是否违反了任何现行有效的 ADR
3. 如果提案与 ADR 发生冲突，判断是需要修改提案，还是应该重新审视该 ADR
4. 将你的发现连同相关 ADR 的具体引用一并列出来
