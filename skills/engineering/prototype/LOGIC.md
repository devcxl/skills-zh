# 逻辑（Logic）

本文件指导 `prototype` 技能如何梳理并描述原型的业务逻辑。

## 从用户故事中提取逻辑

将 PRD 或用户故事转化为具体的、可执行的逻辑。你需要同时描述**界面**和**逻辑**，但本文件重点关注后者——即用户操作时实际发生的事情。

直接从用户故事中提取逻辑。对于用户故事中的每一个"当 / 那么 / 并且"：

- **当** —— 触发器。用户做了什么动作？
- **那么** —— 系统响应。UI 的可见变化是什么？
- **并且** —— 附加的响应或边缘情况。

## 精确描述状态转换

对于原型中的每个交互，你需要说明：

```ts
// 描述类似这样
type PrototypeState = {
  view: "idle" | "loading" | "success" | "error";
  data: DataShape | null;
  error: string | null;
  // ... 根据实际需要添加字段
};
```

为每个状态定义：

- **UI 渲染什么** —— 加载时显示 spinner？空状态如何展示？
- **哪些 action 可用** —— 用户在每种状态下可以做什么？
- **状态转换** —— 哪些 action 或外部事件会触发状态的变化？

## 数据流

描述数据如何在不同的组件和状态之间流转：

- **数据从哪里来** —— 用户输入、API 响应、本地存储？
- **数据如何变化** —— 经过哪些转换步骤？
- **数据流向哪里** —— 最终显示在哪里？

## 与 UI 分离

逻辑层的代码应保持与框架无关。界面组件负责渲染，逻辑层负责决定**渲染什么**和**何时渲染**。原型中使用 `useState` 这样的 hook 是可以的，但复杂的状态机应抽离为纯函数。
