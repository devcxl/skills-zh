# HTML 报告

这是 `improve-codebase-architecture` 的最终交付物。报告会直接写入仓库根目录，命名为 `ARCHITECTURE_REPORT.html`。

## 目的

报告的设计目标是让维护者能在 **3 分钟以内**吸收并做出决定。

## 页面结构

### 头部区域

标题：**代码基架构报告**

副标题：**基于 <仓库名>（最近 <N> 个 issue / <N> 个 PR）的分析**

### 分类标签栏

报告顶部是一条标签栏，每一类问题对应一个标签。点标签会在下方展示对应的详情面板。第一个标签（影响面最大的问题）默认选中。

每个标签包含：issue 标题 + 严重度徽章。

### 详情面板

每个详情面板都是针对某一类问题的一个**可执行规格说明**。结构如下：

#### 摘要（1–2 句）

用 1–2 句话说明问题所在。尽量简短——这里是"电梯演讲"。

#### 问题

描述你观察到的模式——用自然的段落叙述（2–4 段），并引用具体的模块名称和领域术语。

#### 证据

精确列出该模式出现在哪些 issue 和 PR 中，格式如下：

```
- #<number>: PR 标题 / issue 标题
- #<number>: PR 标题 / issue 标题
```

#### 方案

用 2–4 段话，描述建议的整改方向：

- 新增或修改的类型
- 新增或修改的函数签名
- 与现有模块的集成方式
- 在处理该问题时需要遵守的约束

写清楚整改的形态、涉及的接口，以及它如何解决观察到的问题。同时说明为什么这个方案优于备选方案。

#### 示例

##### 当前代码路径

```ts
// 展示有问题的现有代码模式（精简版）
```

##### 建议代码路径

```ts
// 展示相同场景下采用新方案后的代码（精简版）
```

**示例要求**：

- 在当前和建议代码路径中使用完全相同的场景——让对比一目了然
- 两者均控制在 30 行以内
- 优先使用仓库自身的领域术语和实际类型
- 避免含糊的变量名（如 `x`、`data`、`result`）——使用有具体含义的名称

### 页脚

轻量页脚：**生成时间：<时间戳> · <仓库名> · 使用 /improve-codebase-architecture**

## HTML 技术实现

- 使用 `<summary>` / `<details>` 实现折叠区域（而不是自己造轮子）。
- 所有 CSS 内联在 `<style>` 标签中——不引入外部依赖，也不使用构建工具。
- 在缺少 JS 的环境下也可用。如果 JS 可用，用 JS 实现标签切换；如果 JS 不可用，则静态回退到全部展开。
- 配色以灰色为主调，干净中性。
- 使用可用的系统字体。
