# Deepening（深入挖掘）

"deepening" 是一种从近期工作中提炼洞察的方法，目的是找出整个代码基中**重复出现的模式**。

## 第一步：采集 —— 挖掘近期工作

扫描最近合并的 PR（建议回溯 2–4 周），阅读原始 issue 及其评论，了解每项改动**为什么**要做。记录以下信息：

1. 每个 PR 的概要描述
2. 受影响的模块
3. 变更的范围（广泛还是局部？）
4. 本次变更是修复了一个 bug、新增了一个功能，还是重构了一部分代码？

## 第二步：识别 —— 寻找重复模式

在第一步采集的 PR 中寻找共同点。问自己：

- 有没有多个 PR 都在修改同一区域？→ 该区域可能缺乏恰当的抽象。
- 有没有多个 PR 都在处理同一类问题？→ 可能存在尚未被识别的横向关注点。
- 有没有新功能的代码行数很大？→ 也许缺少共享的基础设施或工具函数。
- 在相互关联的区域中，bug 是否反复出现？→ 这些模块可能不够"深"（参见 [deep modules](../tdd/deep-modules.md)）。

将这些模式整理成一个优先队列，优先级的依据是——

1. **重复频率**：该模式出现的次数
2. **影响面**：受影响的模块数量或代码范围
3. **扩散风险**：如果放任不管，将来还会有多少改动受到它的影响

## 第三步：输出（供阶段 3 使用）

生成一份简短的总结，列出 3–5 个最重要的、反复出现的架构模式。

每个模式需包含：

- **描述** —— 你观察到了什么
- **证据** —— 关联哪些 PR / issue
- **当前成本** —— 它正在以什么方式拖慢开发
- **约束** —— 任何已知的不可碰的边界（ADRs、平台限制、性能要求）
