# Language（语言）

这是 `improve-codebase-architecture` 的阶段 5。它的作用是提供一套用词和语气规范，让你在与用户讨论架构变更时，既不会让他们觉得自己被攻击了，也能促成有效的改进。

## 不要指责过去的决策

不要暗示当前的架构"很差"。你审查的代码是由擅长此领域的人在特定约束下编写的。架构之所以成为现在的样子，往往有其原因；问题出在需求变了、规模变了、或者产品走向变了。"

**差：**"当前的架构是一团乱麻。"

**好：**"我看到有几个地方存在重复的模式，可能正在拖慢开发速度。让我带你看看。"

## 不要用"技术债务"这个词

"技术债务"太笼统了。你要做的是具体指出问题所在、它造成了什么影响、以及如何改进。把问题称为"架构机会"或直接描述具体的模式。

**差：**"这个模块的技术债务已经难以承受。"

**好：**"有 3 个 issue 在过去两周内都修改了同一个逻辑。在我们的 HTML 报告里，这里有一个减少重复、降低回归风险的机会。"

## 强调约束，而不是无能

如果一个模式的出现是因为平台、时间或企业政策等硬性约束，直说。这让对话从"你们做错了"转变为"在这些限制条件下，我们还能怎样做得更好"。

**差：**"你怎么没做 X？"

**好：**"考虑到团队用的是 RabbitMQ 而不是 Kafka，现有的模式是合理的。与此同时，有一些轻量的调整可以在不改变消息基础设施的前提下减少重复。"

## 展示而非空谈

始终附上精简的、可对比的代码示例来支撑你的建议。`improve-codebase-architecture` 要求每个建议都要有一个"当前 vs 建议"的代码对比——这条原则同样适用于对话。

## 把"为什么"放在最前面

报告中的每个部分都要以问题陈述开头，以方案结尾。不要跳步骤。在没有说清楚"为什么这是问题"以及"问题的范围是什么"之前，没有人能评估你的方案。
