# 撰写 PRD

围绕用户的自然语言描述，以结构化的调研和提问来充实 PRD。把模糊的想法转化为详实、可审查、无歧义的产品需求文档。你的工作分为三个阶段。

## 流程

### 阶段一：搭建 PRD 大纲

通过提问来搭建 PRD 大纲。抓住用户最初的想法，将其拆解为 PRD 大纲的章节，然后逐一探询：

- **用户故事** —— 谁在用这个功能？他们想要完成什么？为什么？
- **验收标准** —— 如何判断功能已经完成？以可测试的条款列出。
- **边界情况** —— 空输入怎么办？错误如何处理？并发情况怎样？
- **技术约束** —— 平台要求、性能要求、外部依赖。
- **非功能需求** —— 安全性、可访问性、性能。

### 阶段二：审阅、复核、深挖

第二阶段是一轮深入审查（详见 [grill-with-docs](../grill-with-docs/SKILL.md)）。对照项目的 ADR（`docs/adr/`）和领域文档（`CONTEXT.md`）进行审查，找出遗漏的细节，并让用户做出决定。你正在进行的是设计审查，而非格式检查。目的是让 PRD 足够完备，使实现者不必在路上停下来讨要上下文。

### 阶段三：发布到 Issue 跟踪器

一旦 PRD 打磨到位，即将其发布到 issue 跟踪器。在此之前需先获得用户批准。

---

## 原则

- **PRD 是给实现者看的**。用开发者能直接转化为代码的语言来表述验收标准。
- **含糊不清是 bug**。不接受空白空格。如果 PRD 在某个方面存在歧义，问清楚。
- **保持篇幅适中**。PRD 是实际的工作文档，不是研究论文。每个章节都应该对实现有直接帮助。
