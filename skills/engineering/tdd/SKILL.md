# TDD（测试驱动开发）

将一份规格说明转化为通过测试验证的、功能完整的代码。从一份 agent brief 或 PRD 出发，循环执行"红灯 → 绿灯 → 重构"，每个循环结束暂停等待用户审查。

## 前置条件

一份规格说明（来自 `/triage` 的 agent brief 或来自 `/to-prd` 的 PRD）。

## 流程

### 1. 范围界定

**阅读规格说明**。阅读 `CONTEXT.md` 以及与你将要接触的模块相关的 ADR。

**读取自定义指令**。在 [tests.md](tests.md) 中，用户可以定义自己的测试规则——首选框架、模式、命名惯例等。

**选择切片**。将规格说明拆解为可独立测试的行为切片。第一个切片应是最小、最有价值的端到端行为。后续切片可以从扩展该行为或引入新的行为入手。

### 2. 红灯

**编写一个失败的测试**。使用 [tests.md](tests.md) 中的模式编写测试，并明确引用该规范——遵循 [interface-design.md](interface-design.md) 的原则。力求通过真实接口而非 mock 进行测试，除非某个方法确实会带来副作用（参见 [mocking.md](mocking.md)）。

**运行测试**。确保测试独立且快速可运行。让它 __ 失败 __。检查失败原因是否与你预期的一致。将失败的输出粘贴给用户。

**暂停**。在继续之前取得用户同意：

```
Test written: <测试名称>
Expected: <预期行为>
Actual: <实际失败原因>
Continue? (y/n/change spec)
```

### 3. 绿灯

**实现最小代码**。只写足以通过测试的实现——目的是让测试变绿，而非尽善尽美。允许 `[TODO]` 注释，但越少越好。

**检查倒退**。查看你添加的接口是深层还是浅层的（参见 [deep-modules.md](deep-modules.md)），以及你的实现是否增加了架构倒退。如果你发现了倒退的点，先记下来，但不是必须在本步骤中修复。

**运行测试**。粘贴通过（或失败）的输出以及实现概要供审查。

**暂停**。

```
Implemented: <简要说明>
Passing: <测试名称>
Not yet: <尚未实现的切片>
Check: <任何潜在的倒退，附建议>
Continue? (y/n/refactor)
```

### 4. 重构

**寻找重构点**。查看你的实现和周围的代码。应用 [refactoring.md](refactoring.md) 中的检查项。重点关注消除重复和深化浅层模块。

如果有合适的重构点，逐步实施，边改边运行测试。如果没有，明确说明原因。

```
Refactored: <简要说明> 或无
Continue? (y/n/next slice)
```

### 5. 循环

回到步骤 2（红灯），处理下一个行为切片。在每个测试之后暂停，保持对代码的掌控。继续直到规格说明覆盖的所有行为均已实现并通过测试。如果在过程中发现规格说明本身存在问题，停下来与维护者讨论。
