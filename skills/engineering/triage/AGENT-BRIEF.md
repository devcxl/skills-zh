# 撰写 Agent Brief

Agent brief 是在 issue 转移到 `ready-for-agent` 时，发布在 GitHub issue 上的一条结构化评论。它是 AFK agent 开展工作的权威规格说明。原始的 issue 正文和讨论是上下文——agent brief 才是合约。

## 原则

### 耐久性优先于精确性

issue 可能在 `ready-for-agent` 状态下停留数天甚至数周。代码库在此期间会发生变化。撰写 brief 时，要确保即使文件被重命名、移动或重构，它仍然有用。

- **应当**描述接口、类型和行为契约
- **应当**指明 agent 需要查找或修改的具体类型、函数签名或配置结构
- **不要**引用文件路径——这些会过时
- **不要**引用行号
- **不要**假设当前的实现结构会保持不变

### 行为导向，而非过程导向

描述系统**应该做什么**，而非**如何实现**。agent 会以全新的视角探索代码库并自行做出实现决策。

- **好的表述：**"`SkillConfig` 类型应接受一个类型为 `CronExpression` 的可选 `schedule` 字段"
- **差的表述：**"打开 src/types/skill.ts，在第 42 行添加一个 schedule 字段"
- **好的表述：**"当用户不带参数运行 `/triage` 时，应看到一个需要关注的 issue 摘要"
- **差的表述：**"在主处理函数中添加一个 switch 语句"

### 完备的验收标准

agent 需要知道何时算完成。每个 agent brief 都必须包含具体、可测试的验收标准。每一条标准都应能独立验证。

- **好的表述：**"运行 `gh issue list --label needs-triage` 返回的是已经过初步分类的 issue"
- **差的表述：**"分类功能应该正常工作"

### 明确的范围边界

说明哪些不在范围内。这可以防止 agent 过度优化或对相邻功能做不当假设。

## 模板

```markdown
## Agent Brief

**类别：** bug / enhancement
**摘要：** 用一句话描述需要做什么

**当前行为：**
描述现状。对于 bug，即出错的行为。
对于 enhancement，即该功能建立于其上的基准状态。

**期望行为：**
描述 agent 工作完成后系统应有的表现。
对边界情况和错误条件要具体说明。

**关键接口：**
- `TypeName` — 需要做什么变更及其原因
- `functionName()` 返回类型 — 当前返回什么 vs 应该返回什么
- 配置结构 — 任何需要新增的配置选项

**验收标准：**
- [ ] 具体、可测试的标准 1
- [ ] 具体、可测试的标准 2
- [ ] 具体、可测试的标准 3

**不在范围内：**
- 不应在本 issue 中变更或处理的事项
- 看似相关但实际上独立的功能
```

## 示例

### 好的 agent brief（bug）

```markdown
## Agent Brief

**类别：** bug
**摘要：** 技能描述在截断时会在词中切断，导致输出异常

**当前行为：**
当技能描述超过 1024 个字符时，会严格在 1024 字符处截断，不考虑词的边界。
这导致描述在词的中间被截断（例如 "Use when the user wants to confi"）。

**期望行为：**
截断应在 1024 字符前的最后一个词边界处进行，并追加 "..." 表示截断。

**关键接口：**
- `SkillMetadata` 类型的 `description` 字段 — 类型无需更改，
  但填充该字段的验证/处理逻辑需要尊重词边界
- 任何读取 SKILL.md frontmatter 并提取 description 的函数

**验收标准：**
- [ ] 低于 1024 字符的描述保持不变
- [ ] 超过 1024 字符的描述在 1024 字符前的最后一个词边界处截断
- [ ] 被截断的描述以 "..." 结尾
- [ ] 含 "..." 的总长度不超过 1024 字符

**不在范围内：**
- 改变 1024 字符限制本身
- 多行描述支持
```

### 好的 agent brief（enhancement）

```markdown
## Agent Brief

**类别：** enhancement
**摘要：** 支持 `.out-of-scope/` 目录，用于跟踪被拒绝的功能请求

**当前行为：**
当功能请求被拒绝时，issue 以 `wontfix` 标签和一条评论关闭。没有对决策或理由的持久记录。未来出现类似的请求时，维护者需要回忆或搜索以往的讨论。

**期望行为：**
被拒绝的功能请求应记录在 `.out-of-scope/<concept>.md` 文件中，该文件应包含决策、理由以及所有请求过该功能的 issue 链接。在对新 issue 进行分类时，应检查这些文件是否有匹配项。

**关键接口：**
- `.out-of-scope/` 中的 Markdown 文件格式 — 每个文件应包含
  `# 概念名称` 标题、`**决策：**` 行、`**理由：**` 行
  以及 `**过往请求：**` 列表（带 issue 链接）
- 分类流程应在早期读取所有 `.out-of-scope/*.md` 文件，
  并通过概念相似度将收到的 issue 与这些文件进行匹配

**验收标准：**
- [ ] 以 wontfix 关闭功能请求时，会创建/更新 `.out-of-scope/` 中的文件
- [ ] 文件包含决策、理由以及已关闭 issue 的链接
- [ ] 如果匹配的 `.out-of-scope/` 文件已存在，新的 issue
      会追加到该文件的"过往请求"列表中，而非创建重复文件
- [ ] 在分类时，会检查现有的 `.out-of-scope/` 文件，
      并在新 issue 匹配到过往拒绝记录时予以提示

**不在范围内：**
- 自动匹配（由人工确认匹配）
- 重新打开之前被拒绝的功能
- Bug 报告（仅 enhancement 拒绝进入 `.out-of-scope/`）
```

### 差的 agent brief

```markdown
## Agent Brief

**摘要：** 修复分类 bug

**要做的事：**
分类功能出了点问题。看看主文件把它修好。
大约在第 150 行的函数有问题。

**要改的文件：**
- src/triage/handler.ts（第 150 行）
- src/types.ts（第 42 行）
```

为什么差：
- 没有类别
- 描述模糊（"分类功能出了点问题"）
- 引用了会过时的文件路径和行号
- 没有验收标准
- 没有范围边界
- 没有描述当前行为 vs 期望行为
