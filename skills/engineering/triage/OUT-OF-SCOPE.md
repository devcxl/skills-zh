# Out-of-Scope 知识库

仓库中的 `.out-of-scope/` 目录用于存储被拒绝功能请求的持久记录。它有两个作用：

1. **机构记忆** —— 记录功能被拒绝的原因，让理由不会随着 issue 关闭而丢失
2. **去重** —— 当新提交的 issue 与之前被拒绝的请求匹配时，技能可以直接呈现过往决策，而不是重新讨论

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

按**概念**分文件，而非按 issue。多个请求同一事项的 issue 归入同一个文件。

## 文件格式

文件应采用轻松、可读的风格撰写——更像一份简短的设计文档而非数据库条目。使用段落、代码示例和实例来让推理清晰，让第一次看到的人也能理解。

```markdown
# 深色模式

本项目不支持深色模式或面向用户的自定义主题。

## 为什么不在范围内

渲染管线假定在 `ThemeConfig` 中定义了一套单一的色彩方案。
支持多主题需要：

- 一个包裹整个组件树的主题上下文 provider
- 每个组件都需要支持主题感知的样式解析
- 一个用于持久化用户主题偏好的存储层

这是一项重大的架构变更，与项目聚焦内容创作的定位不一致。
主题化应由下游消费方（嵌入或再分发输出的使用者）来关注。

```ts
// 当前的 ThemeConfig 接口并非为运行时切换而设计：
interface ThemeConfig {
  colors: ColorPalette; // 单一色彩方案，构建时解析
  fonts: FontStack;
}
```

## 过往请求

- #42 — "支持深色模式"
- #87 — "无障碍夜间主题"
- #134 — "深色主题选项"
```

### 文件命名

为概念使用简短、描述性的 kebab-case 名称：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应足够直观，让浏览目录的人不必打开文件就能理解被拒绝的是什么。

### 撰写理由

理由要有实质内容——不是"我们不想做"，而是**为什么**。好的理由会引用：

- 项目范围或理念（"本项目聚焦于 X；主题化是下游消费方的关注点"）
- 技术约束（"支持此功能需要 Y，这与我们的 Z 架构冲突"）
- 战略决策（"我们选择使用 A 而非 B，因为……"）

理由应具备持久性。避免引用临时情况（"我们现在太忙"）——那不是真正的拒绝，而是推迟。

## 何时检查 `.out-of-scope/`

在分类过程中（步骤 1：收集上下文），读取 `.out-of-scope/` 中的所有文件。评估新 issue 时：

- 检查请求是否匹配已有的 out-of-scope 概念
- 匹配依据是概念相似度而非关键词——"夜间主题"匹配 `dark-mode.md`
- 如果存在匹配，呈现给维护者："这与 `.out-of-scope/dark-mode.md` 类似——我们之前因为 [理由] 拒绝了这个请求。你现在仍持相同看法吗？"

维护者可以：

- **确认** —— 新 issue 追加到已有文件的"过往请求"列表中，然后关闭
- **重新考虑** —— 删除或更新 out-of-scope 文件，issue 进入正常分类流程
- **不同意** —— 两个 issue 相关但不同，继续正常分类流程

## 何时写入 `.out-of-scope/`

仅当某个 **enhancement**（而非 bug）被拒绝为 `wontfix` 时。流程为：

1. 维护者决定某个功能请求不在范围内
2. 检查是否已有匹配的 `.out-of-scope/` 文件
3. 如果有：将新 issue 追加到"过往请求"列表
4. 如果无：创建新文件，包含概念名称、决策、理由和第一条过往请求
5. 在 issue 中发布评论，解释决策并提及 `.out-of-scope/` 文件
6. 以 `wontfix` 标签关闭 issue

## 更新或移除 out-of-scope 文件

如果维护者改变了对之前被拒绝概念的看法：

- 删除对应的 `.out-of-scope/` 文件
- 技能不需要重新打开旧 issue——它们是历史记录
- 触发了重新考虑的新 issue 进入正常分类流程
