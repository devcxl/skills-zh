# ADR 格式

ADR 存放在 `docs/adr/` 目录，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒加载创建 `docs/adr/` 目录——仅当第一个 ADR 需要时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们做了什么决策，以及为什么。}
```

就是这样。一个 ADR 可以只有一段。其价值在于记录*做了*决策以及*为什么*——而不是填充章节。

## 可选章节

仅当这些章节能提供真正价值时才包含它们。大多数 ADR 不需要它们。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **Considered Options**（考虑的选项）——仅当被否决的替代方案值得记住时
- **Consequences**（后果）——仅当非显而易见的下游影响需要指出时

## 编号

扫描 `docs/adr/` 找到已有的最高编号，然后加一。

## 何时提供 ADR

以下三个条件必须同时满足：

1. **难以逆转**——事后改变想法的成本很高
2. **脱离上下文会令人困惑**——未来的读者看到代码会想"他们到底为什么这样做？"
3. **是真实权衡的结果**——存在真正的替代方案，你因特定原因选择了其中一个

如果决策容易逆转，跳过它——你将来会把它改回来的。如果不会令人困惑，没人会质疑为什么。如果没有真正的替代方案，除了"我们做了显而易见的事"之外也没什么可记录的。

### 什么算作 ADR 的内容

- **架构形态。** "我们使用 monorepo。" "write model 是 event-sourced 的，read model 投射到 Postgres 中。"
- **context 之间的集成模式。** "Ordering 和 Billing 通过 domain events 通信，而非同步 HTTP。"
- **带有锁定效应的技术选型。** 数据库、消息总线、auth 提供方、部署目标。不是每个 library——只是那些更换起来需要一季度的。
- **边界和范围决策。** "Customer 数据由 Customer context 拥有；其他 context 仅通过 ID 引用它。" 明确的否定和肯定一样有价值。
- **有意偏离常规路径。** "我们因为 X 原因使用手动 SQL 而非 ORM。" 任何会让理性的读者假设相反做法的事情。这些可以阻止下一位工程师"修正"某些有意为之的设计。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。" "由于 partner API 合约，响应时间必须在 200ms 以内。"
- **被否决的替代方案（当否决原因不显而易见时）。** 如果你考虑了 GraphQL 但因微妙的原因选择了 REST，记录下来——否则六个月后又会有人建议用 GraphQL。
