# 编写文档页面

`engineering/` 和 `productivity/` 中的每项技能在 `docs/<bucket>/<skill-name>.md` 都有一个面向人类的**文档页面**——文档树镜像了 `skills/` 下的这两个 bucket 文件夹。它发布在 `https://aihero.dev/skills-<skill-name>`；URL 始终是 `skills-<skill-name>`，无论 bucket 如何，因此文档路径仅用于仓库内部组织。该页面不是技能本身，也不是 `SKILL.md` 的副本。只有这两个 bucket 被推广；其余部分（`misc/`、`personal/`、`in-progress/`、`deprecated/`）不提供文档页面。

这些技能大多数是**由用户调用的**：代理永远不会为你触发它们，因此*你*是需要记住它们存在以及何时使用它们的索引。那份记忆就是**认知负载**。文档页面的工作就是缓解它——让读者围绕一项技能建立认知，以便他们能在脑内记住它，知道何时使用它，并看到它在整个系统中的位置。这些页面共同构成一个分布式的路由器；每个页面都是一个节点。

当一项被推广的技能被添加、重命名或行为发生变更时，执行以下操作：创建或重新同步其文档页面。重命名也会移动文件（`docs/<bucket>/<old>.md` → `docs/<bucket>/<new>.md`），因为发布的 URL 跟随技能名称；一项在 `engineering/` 和 `productivity/` 之间移动的技能，其文档文件也会移动到对应的文件夹。`misc/`、`personal/`、`in-progress/` 和 `deprecated/` 中的技能没有页面——这些 bucket 都不被推广。一项技能*从*这些 bucket 之一移入 `engineering/` 或 `productivity/`，则获得页面；向相反方向移动则失去页面。

由于这些页面发布在 `aihero.dev` 上，**每个链接都是绝对路径**——绝不使用仓库相对路径。指向其他技能的链接指向 `https://aihero.dev/skills-<name>`；指向仓库的链接使用完整的 `https://github.com/mattpocock/skills/...` URL。在仓库中有效的相对链接一旦发布就会失效。

没有 H1——发布后的页面从 slug 中获取标题。

## 它的作用

用一两段通俗易懂的语言描述。以技能的一句话职责开头，然后陈述**定义性约束**——使这个技能与显而易见的默认行为不同的关键事实（对于 `to-prd`：它不会再次盘问用户，它综合已经确定的信息）。用一句简单的陈述句来写——永远不要用像"定义性约束："或"关键点："这样标注出来的旁白；这种句式读起来像填充内容。这行是页面上最有价值的内容；永远不要省略。

## 何时使用

如何以及何时使用该技能——两拍，实际上始终存在：

- **调用模式。** 说明是你输入还是代理触发它。用户调用的技能："你通过键入 `/<name>` 来调用它——代理不会主动使用它。"模型调用的技能："键入 `/<name>`，或者当任务适合时代理会自动使用它。"
- **触发边界。** 索引条目："当……时使用它"。当技能与兄弟技能容易混淆时，加上另一半——"对于 <X>，使用 [<sibling>](https://aihero.dev/skills-<sibling>) 代替。"

## 前置条件

可选——仅当技能需要某些条件就位才能运行时包含；否则完全省略该标题。涵盖：**它写入的工作区**（像 `grill-with-docs` 这样的有状态技能写入 `CONTEXT.md` 和 ADR；`teach` 构建一个完整的目录——说明它写入什么以及在哪里）、**前置设置**（`triage`/`to-prd`/`to-issues` 需要 `setup-matt-pocock-skills` 已配置工单跟踪器）、或**仓库特定工具**。可以在任何位置运行的无状态技能没有前置条件——删除此部分。

## <自由形式中部>

一到三个简短部分，使用技能*自己的词汇*，使其豁然开朗——选择适合该技能的任何标题：它运行的循环、它产生的产物、它做出的分叉、它消除的一个反模式。没有预设的标题；技能之间差异太大，无法统一。

唯一不可协商的是：**展示技能的引导词汇/定义性概念**——tight 反馈循环、deep module、一次性代码回答问题、red-green。它产生双重回报：读者了解技能是*什么*，并了解他们以后会用来*想起*它的那个词汇。

## 它发挥作用的表现

可选。一个简短、可检查的列表，列出可观察的信号，告知读者技能是否在真正发挥作用——当它触发时应该看到什么，以及通过缺失来判断它尚未触发。当技能有明确的标志时包含它（例如 `to-prd` 在不重新盘问你的情况下编写文档；一个引导词汇在追踪中重新出现）；当信号模糊时省略此标题。几个要点，不超过。

## 它的定位

始终存在。用一两句话将技能定位在系统中：

- **角色。** 命名它：**链步骤**（`grill-with-docs → to-prd → to-issues → implement → code-review`）、**一次性设置**（`setup-matt-pocock-skills`）、**定期维护**（`improve-codebase-architecture`，“每几天”）、或**可随时使用的独立工具**（`diagnosing-bugs`、`prototype`、`handoff`）。独立工具的地图用一句诚实的话即可——远胜于省略此部分。
- **邻居。** 一个或两个重要的兄弟技能，每个都带有一个 because 从句，使用绝对链接。
- **地图。** 指向 [ask-matt](https://aihero.dev/skills-ask-matt)（跨越整个技能集的路由器），这样这个页面保持为一个节点，永远不需要重新绘制整个图。

## 约定

- 解释**为什么**，而不是过程。页面定位技能并说明其适用场景；它从不复现 `SKILL.md` 中的步骤或模板转储——选择工具的人类不需要操作手册。
- 使用技能的**引导词汇**（_seam_、_deep module_、_tracer bullet_），使页面和技能讲同一种语言。
- 保持页面本身的低负载。它是关于*低认知负载*技能的文档；装饰（多余的标题、重复的链接）正是它所要反对的东西。

## 完成标准

- 页面存在于 `docs/<bucket>/<name>.md`，并且在重命名或 bucket 移动后没有残留的旧页面。
- Quickstart 块和来源链接命名了正确的 bucket 和技能；更新命令行命名了技能名称。
- `## 它的作用` 陈述了定义性约束，以平实的散文而非标注出来的旁白形式呈现。
- `## 何时使用` 陈述了调用模式和触发边界。
- `## 它的定位` 命名了角色并链接到 `ask-matt`。
- 前置条件（工作区、前置设置、工具）在存在时被说明，不存在时该部分被省略。
- 中部展示了引导词汇。
- 每个链接都是绝对的，并且每个链接都能解析。
